## 1. 系统概述

### 1.1 硬件平台

| 组件 | 型号 | 接口 |
|------|------|------|
| MCU | STM32WLE5JCI6 | Cortex-M4 @ 48MHz |
| 温湿度传感器 | SHT40 | I2C1 (PB7/PB6) |
| CO 传感器 | PS-CO-5000 | USART2 (PA3/PA2), 9600bps |
| GNSS 模块 | ATGM336H | USART1 (PA10/PA9), 115200bps |
| EEPROM | BL24C256A (32KB) | I2C1 (与 SHT40 同总线) |
| OLED 显示屏 | SSD1306 128x64 | I2C2 (PB9/PB8) |
| 按键 | 3 个 (UP/OK/DOWN) | PB12/PB2/PA15, EXTI 触发 |
| LED 指示灯 | 1 个 | PC0, 高电平有效 |
| 电源控制 | 3 路 (ADC_EN/5V_EN/GNSS_EN) | PB5/PB3/PA8 |

### 1.2 软件概览

终端节点是一个完整的嵌入式传感器数据采集与上报系统。核心功能包括：

- **传感器采集**: 定期读取 SHT40 (温湿度) + PS-CO-5000 (一氧化碳) + 电池电压
- **火灾预警判定**: 基于温度/CO/湿度阈值的三级运行模式切换 (常规/预警/火警)
- **GNSS 定位**: 支持稳定性过滤的 GPS/北斗定位，结果持久化到 EEPROM
- **LoRa 组网**: 完整的私有协议栈 (注册/Beacon 扫描/父节点选择/端到端 ACK/指令下发)
- **OLED 菜单**: 128x64 屏显示实时数据 + 参数配置界面
- **低功耗**: STOP2 深度休眠 + WFI 浅睡双级管理，RTC 闹钟+按键唤醒
- **离线缓存**: 通信中断时 EEPROM 缓存传感器数据，恢复后自动补发

## 3. 主状态机

### 3.1 状态定义

```c
typedef enum {
    STATE_INIT,              // 系统初始化
    STATE_IDLE,              // 空闲待机 (轮询采集/发送时机)
    STATE_SETTINGS,          // 设置模式 (OLED 菜单操作)
    STATE_MEASURE,           // 传感器采集
    STATE_GNSS,              // GNSS 定位
    STATE_TRANSMIT,          // LoRa 发送
    STATE_RECEIVE,           // LoRa Class A RX 窗口
    STATE_SLEEP,             // 低功耗休眠
    STATE_ERROR,             // 错误处理
    STATE_PROTO_SCAN,        // 协议: Beacon 扫描
    STATE_PROTO_REGISTERING, // 协议: 注册
} app_state_t;
```

### 3.2 状态流转图

```
INIT → [有未存位置] → GNSS
   ↓                      ↓ 定位成功
   └──→ SCAN(45s) ←────────┘
          ↓ 选中父节点
       REGISTERING ──→ 注册成功 ──→ IDLE ⇄ SLEEP
          ↓ 超时                       ↓    ↑
          └──→ SCAN(换候选)       MEASURE   │
                                    ↓       │
                                 TRANSMIT   │
                                    ↓       │
                                 RECEIVE ──┘
```

### 3.3 核心流程伪代码

```
app_init():
    state_init_enter()          // 初始化所有模块
    注册按键回调
    清除持久化短地址 (调试模式)
    if 没有位置数据:
        → STATE_GNSS           // 先获取 GNSS
    else:
        → STATE_PROTO_SCAN     // 已有位置则直接组网

app_process():  // 主循环 while(1) 中调用
    board_led_process()          // LED 闪烁处理
    co_process()                 // CO 传感器数据处理
    process_button_events()      // 按键事件分发
    switch current_state:
        case STATE_IDLE:         // 判断是否该采集/发送/休眠
        case STATE_MEASURE:      // do_measure() → update_run_mode()
        case STATE_TRANSMIT:     // do_transmit() → 等待 TX 完成
        case STATE_RECEIVE:      // RX1(2s) → RX2(2s) → 处理 ACK
        case STATE_SLEEP:        // WFI/STOP2 → RTC 闹钟唤醒
        case STATE_GNSS:         // gnss_process() → 等待定位
        case STATE_PROTO_SCAN:   // 45s 扫描 Beacon → 选父节点
        case STATE_PROTO_REGISTERING: // 注册帧 → 等待分配
        ...
    if 状态变化: 调用对应的 _enter() 函数
```

### 3.4 运行模式判定

终端根据传感器数据自动切换运行模式，影响采集/发送频率和休眠策略：

```
传感器数据 → update_run_mode():
    if 温度 > temp_alarm_threshold:
        → FIRE_ALARM (火警)   // 最快频率, 不休眠
    elif CO > co_warning_threshold 或 湿度 < hum_warning_threshold:
        → EARLY_WARNING (预警) // 中等频率
    else:
        → NORMAL (常规)        // 最低频率, 可休眠

每种模式有独立的采集间隔和发送间隔，含迟滞 (触发/恢复阈值不同)
```

## 4. LoRa 通信协议

> 详细协议规范见 `LoRa私有协议_修订版.md`，本节仅概述终端侧实现要点。

### 4.1 帧结构

```
[FC 1B] [DestAddr 2B LE] [SenderAddr 2B LE] [MsgSeq 2B LE] [TLV Payload 可变] [CRC 2B 硬件]
```

TLV: `[Type 1B] [Length 1B] [Value Length字节]`

终端单帧载荷上限 **120 字节**。正常数据帧组装顺序：

```
帧头7B → 0x01实时11B → extra(0x0F配置/0xFF状态) → 0x10历史×N(填满余量)
```

### 4.2 注册流程

```
1. GNSS 定位: 卫星≥6, HDOP<2.0, 连续5次漂移<10m → 保存到 EEPROM
2. SCAN: 主 SF 连续接收 45s, 收集 Beacon 候选列表
3. 父节点评分: IsValid>直连网关>短路经>低负载>强RSSI
4. 发送注册帧: FC=0xC0, Sender=0xFFFF
   Payload: 0x30(12B UID) + 0x31(3B能力) + 0x34(11B GNSS) + 0x35(1B角色)
5. 等待分配:
   - 直连: 15s RX 窗口
   - 中继代理 ACK: 延长至 3min
   - 超时: 排除当前候选 → 回 SCAN
6. 注册成功: ACK 中提取 0x32(短地址) → 写入 RAM (调试模式不持久化)
```

### 4.3 数据帧上行

```c
do_transmit():
    // 1. 构建传感器数据
    sd = {温度×10, 湿度×10, CO浓度, 电池mV, 报警标志}

    // 2. 构建额外 TLV (条件触发)
    if cfg_report_pending:     // 0x29 触发
        extra += proto_tlv_config_report(...)    // 0x0F, 26B
    if cmd_status_count > 0:   // 指令执行后触发
        extra += TLV_CMD_STATUS 队列            // 0xFF, 2+2N B

    // 3. 构建完整帧
    frame = proto_build_data_frame(buf, sensor, parent, sender, seq,
                                   sf_flag, power, extra, extra_len)
    // 内部: 0x01(11B) → extra → 0x10×N (遍历重传队列填充余量)

    // 4. 配置 Radio 并发送
    app_lora_configure(sf, dbm)
    app_lora_send(frame, len)

    // 5. 推入重传队列
    proto_retransmit_push(entry)
```

### 4.4 ACK 与指令处理

```c
handle_received_frame(buf, len, rssi, snr):
    act = proto_dispatch_rx(buf, len, rssi, snr)

    case PROTO_RX_ACK_CONFIRMED:
        proto_retransmit_ack(ack.last_in_seq)   // 移除已确认条目
        execute_dl_commands(buf, len)           // 执行附带的下行指令

        // 重传队列空 → 逐条排出 EEPROM 缓存继续补发
        if proto_retransmit_count() == 0:
            cache_record_t cached
            if app_cache_pop(&cached):
                // 赋予新序列号, 推入重传队列
                proto_retransmit_push(hist)

    case PROTO_RX_PROXY_ACK:
        execute_dl_commands(buf, len)           // 代理 ACK 也可能带指令

    case PROTO_RX_REGISTERED:
        app_config_set_short_addr_ram(addr)     // 仅写 RAM
        registration_done = true
```

### 4.5 下行指令执行

```c
execute_one_command(cmd):
    switch cmd->type:
        0x20 SET_INTERVAL:     更新采集间隔 ×3
        0x21 SET_TX_INTERVAL:  更新发送间隔 ×3
        0x22 SET_TEMP:         更新温度火警阈值 + 恢复
        0x23 SET_CO:           更新 CO 预警阈值 + 恢复
        0x24 SET_HUMIDITY:     更新湿度预警阈值 + 恢复
        0x25 SET_RF:           更新 SF + 功率档位 → 立即生效
        0x26 REPORT_NOW:       立即上报 (设置标志)
        0x27 REQ_RESEND:       请求补发 (TODO: 从 EEPROM 取)
        0x28 UPDATE_PARENT:    切换父节点
        0x29 QUERY_CONFIG:     设置 cfg_report_pending = true
        0x2A REMOTE_RESET:     延迟 5s 复位

    每次执行后: 推入 cmd_status_queue(type+status), 最多 8 条
    下次上行: 队列编码为单个 0xFF TLV (Length = 2×N)
```

## 5. 传感器采集

### 5.1 采集流程

```
do_measure():
    // 1. SHT40 (温湿度) — 含重试
    for retry in 0..2:
        if sht40_read(&data, HIGH_PRECISION):
            更新 temperature/humidity/sht40_valid
            break
        否则 sht40_reset(), 延迟 5ms 重试

    // 2. PS-CO-5000 (一氧化碳)
    co = co_get_data()
    if co.valid && !co.warming_up:
        co_concentration = co.concentration, co_valid = true

    // 3. 电池电压 (ADC)
    battery_voltage = board_adc_read_battery()   // 含 VREFINT 校准

    // 4. 更新运行模式
    update_run_mode()  // 含迟滞判断

    最后: last_measure_tick = HAL_GetTick()
```

### 5.2 CO 传感器管理

PS-CO-5000 上电后需 **30 秒** 稳定。驱动程序通过状态机管理:

```
CO_STATE_OFF → co_start() → CO_STATE_WARMUP (30s)
                            → CO_STATE_ACTIVE (数据有效)
```

- 等待期间 `co_data.warming_up = true`, 采集时跳过
- ACTIVE 后若 **5 秒** 无有效帧 → `co_data.valid = false`
- CO 传感器与调试串口共享 USART2 (9600bps), DMA Circular 接收

### 5.3 报警判定逻辑

```
温度火警: T > temp_alarm_threshold        → FIRE_ALARM (最高优先级)
          (火警模式下 T > temp_alarm_recovery 才维持)

CO预警:   CO > co_warning_threshold       → EARLY_WARNING
          (预警模式下 CO > co_warning_recovery 才维持)

湿度预警: H < hum_warning_threshold       → EARLY_WARNING
          (预警模式下 H < hum_warning_recovery 才维持)

报警标志位: ALARM_FLAG_TEMPERATURE(0x04) | CO(0x01) | HUMIDITY(0x02)
```

## 6. GNSS 定位

### 6.1 驱动架构

- ATGM336H 通过 USART1 (115200bps) 输出 NMEA-0183 语句
- DMA Circular 模式接收 (512B 缓冲), 空闲中断通知有新数据
- 解析 GGA (经纬度/HDOP/卫星数) + RMC (状态验证)

### 6.2 稳定性过滤

定位需满足 **三层条件** 才进入 FIXED 状态:

| 层级 | 条件 | 常量 |
|------|------|------|
| 卫星数 | ≥ 6 颗 | `GNSS_MIN_SATELLITES` |
| HDOP | < 2.0 (×10 < 20) | `GNSS_MAX_HDOP_X10` |
| 连续稳定 | 5 次位置漂移 < 10m | `GNSS_STABLE_COUNT` / `GNSS_POS_DRIFT_THRESHOLD(55≈10m)` |

```
gnss_process() SEARCHING 状态:
    质量不达标 → stable_count = 0
    质量达标:
        stable_count == 0 → 记录参考位置, count=1
        is_position_stable():
            偏差 < 55 (≈10m) → count++, count≥5 → FIXED
            偏差过大 → count=0 (下一轮重新记录)

总超时 120s: SEARCHING → TIMEOUT → 重启模块重试
```

## 7. 配置存储与缓存

### 7.1 EEPROM 地址分配

```
0x0000 - 0x00FF: 参数区 (256B)  — app_config_t (~100B, 含校验)
0x0100 - 0x011F: 位置数据区 (32B) — gnss_stored_location_t
0x0120 - 0x012F: 缓存管理区 (16B) — cache_mgmt_t (head/tail/count)
0x0130 - 0x7FFF: 缓存数据区 (~32KB) — cache_record_t 环形队列
```

### 7.2 参数结构

```c
app_config_t:
    magic, version, uid_hash(Crc32 of UID)
    short_id, measure_interval_[normal/warning/alarm],
    transmit_interval_[normal/warning/alarm],
    temp_alarm_threshold/recovery (float, °C),
    co_warning_threshold/recovery (float, ppm),
    hum_warning_threshold/recovery (float, %RH),
    lora_tx_power, lora_sf (旧字段, 保留),
    proto_short_addr, proto_parent_addr, proto_power_level,
    proto_sf_primary, proto_sf_backup, proto_sf_current,
    proto_msg_seq, proto_has_short_addr
```

### 7.3 重传与缓存机制

```
重传队列 (RAM):
    容量 16 条 (proto_retransmit_entry_t, 含 msg_seq)
    do_transmit() 推入 → ACK LastInSeq 移除

EEPROM 缓存 (环形队列):
    写入: TX 失败 或 ACK RX 超时 → app_cache_push()
    排出: ACK 收到且重传队列空 → app_cache_pop() → 推入重传队列 → 下次附在 0x10 补发
```

## 9. 低功耗管理

### 9.1 双级休眠策略

| 级别 | 模式 | 功耗 | 唤醒 | 适用场景 |
|------|------|------|------|---------|
| 浅睡 | WFI (CPU 暂停 + 外设门控) | ~1-2 mA | 任意中断 (μs) | 短等待 (≤5s)、CO 预热、RX 窗口间隙 |
| 深睡 | STOP2 (仅 RTC+唤醒 GPIO 供电) | ~1 μA | RTC 闹钟 / 按键 EXTI | 长间隔 (>5s) |

### 9.2 STOP2 进入/退出

```
enter_stop2():
    关闭: ADC_EN, 5V_EN, GNSS_EN, LED
    Radio.Sleep()
    OLED_DisPlay_Off()
    关闭时钟: GPIOC, GPIOH, USART1, I2C1, I2C2, ADC, TIM16
    保留: GPIOA (PA15=BTN_DOWN), GPIOB (PB2=OK, PB12=UP)
    HAL_PWREx_EnableInternalWakeUpLine()  // 使能 RTC 闹钟唤醒
    HAL_SuspendTick()
    HAL_PWREx_EnterSTOP2Mode(PWR_STOPENTRY_WFI)

exit_stop2():
    HAL_ResumeTick()
    重新使能所有时钟
    HAL_I2C_Init(&hi2c1), HAL_I2C_Init(&hi2c2)
    OLED_Init(), OLED_DisPlay_On()
    board_led_set(LED_BLINK_SLOW)
```

### 9.3 CO 预热与休眠协调

```
state_sleep_enter():
    计算下次采集时间 T_meas
    if CO 已关 && 是采集事件:
        if 间隔 > 30s:
            sleep_co_warmup = true
            RTC 闹钟设为 T_meas - 30s  // 提前唤醒

state_sleep_process():
    if sleep_co_warmup:
        co_start()          // 给 CO 上电
        WFI 循环 30s        // 等待预热 (按键仍可响应)
        return STATE_MEASURE
```

### 9.4 不休眠条件

- **火警模式**: 最高优先级，必须持续监控
- **屏幕亮起**: 工作人员正在查看/操作 OLED
- **设置页面**: 正在编辑配置参数

## 10. 已规划未实现的功能

### 10.1 短地址持久化

**状态**: 已规划，代码就绪

**现状**: 调试阶段注册成功后调用 `app_config_set_short_addr_ram()` 仅写 RAM，`app_config_set_short_addr()` 已实现完整 EEPROM 写入但未启用。`app_init()` 中 `app_config_clear_short_addr()` 每此启动强制清除。

**改为部署模式**: 将 `app_config_set_short_addr_ram()` 替换为 `app_config_set_short_addr()`，移除 `app_config_clear_short_addr()` 调用，启动时检查 `app_config_has_short_addr()` 若为 true 则跳过注册直接进入 IDLE。

### 10.2 下行指令联调测试

**状态**: 已实现，未经硬件联调

**现状**: 所有 11 条下行指令 (0x20-0x2A) 的解析和执行逻辑已完整实现，指令结果通过 0xFF 上报。待与网关/中继联调验证。

**可能需要的改进**: 部分指令执行后可能需要在下次上行中增加特定反馈 (如 0x27 补发请求后应查询 EEPROM 并附历史块)。

### 10.3 中文字库补充

**状态**: 缺失 6 个字模

**缺失字**: 短、地、址、父、节、点

**影响**: 主页通过 `draw_str()` → `font16x16` 渲染，缺失字显示为空白 (功能正常)；设备信息页已改用 `draw_str()`，补全字模后即可正常显示。

**添加方法**: 使用波特律动 LED 取模工具 (16x16) 生成字模数据，追加到 `font.c` 的 `zh16x16[]` 数组末尾，更新数组长度。

### 10.4 间隔单位切换 (秒→分钟)

**状态**: 已规划

**说明**: 实际部署后采集/发送间隔将长达数十分钟至数小时。需同步修改:

1. 配置默认值 (`app_config.h` 中 DEFAULT_*_INTERVAL_* 宏)
2. 菜单编辑步长和范围 (长按 UP/DOWN 实现连续加减, 支持切换单位)
3. 休眠计算逻辑 (当前按秒计算，改分后需检查 STOP2 最长休眠时间限制)
4. 协议 `TLV_CONFIG_REPORT (0x0F)` 中字段语义 (保持秒还是改为分钟需与网关协商)

### 10.5 自适应链路控制

**状态**: 部分实现

**现状**: 协议层预留了 SF 切换和功率调节的机制 (`PROTO_LOSS_THRESHOLD`、`PROTO_RECOVER_THRESHOLD`、`PROTO_MAX_CONSECUTIVE_LOSS`)，ACK 中已携带 RSSI/SNR/TxPower/SuggestPower。但主状态机中未实际调用自适应逻辑 (`consecutive_loss` 计数器未更新、SF 切换未触发、功率建议未应用)。

**实现方向**: 在 `handle_received_frame()` 的 ACK 处理中更新 `consecutive_loss` 计数器，在数据帧成功/失败时增减，达到阈值时切换 SF 或调节功率档位，更新 `proto_sf_current` 并调用 `app_lora_configure()`。

### 10.6 系统看门狗 (IWDG)

**状态**: 未实现

**说明**: 部署后需防止野外环境下的固件死锁。STM32WLE5 内置独立看门狗，需在 STOP2 休眠期间处理喂狗策略 (STOP2 下 IWDG 继续运行，需周期性短唤醒喂狗或使用窗口看门狗协调)。

### 10.7 远程固件升级 (FOTA)

**状态**: 未规划

**说明**: LoRa 带宽极低 (SF7 下 ~5kbps)，单帧 120B，传输完整固件 (~100KB+) 需分段传输方案、差分升级或仅下发配置参数。优先级较低，可作为未来增强项。
